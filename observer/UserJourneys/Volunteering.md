# User Journey: Signed‑In User Volunteers for a Project

## Target Audience
- Primary persona: 22-year-old YSA male who is looking for a service project to do. Isn't looking for a long term commitment.

## User’s Context & End Goal
- **Context:** User is already signed in with their account.
- **End Goal:** User successfully commits to volunteer for the project and clearly understands what happens next (where to go, when to show up, and who to contact).

## Journey Stages
1. Browses projects
2. Reviews opportunity details
3. Commits to volunteer
4. Receives confirmation and next steps

## Actions (Action → Result)
1. User presses the arrows on the home page, viewing several projects 
   * System displays several different projects.
2. User opens a project’s detail page 
   * System displays full project information (description, date/time, location, requirements, and organizer details).
3. User selects “Volunteer” (or equivalent CTA)
   * System displays a signup/commitment panel with required fields and capacity status. **[NEEDS VERIFICATION]**
4. User fills out the fields and submits the signup 
   * System records the volunteer commitment and displays a confirmation state with status and next steps. **[NEEDS VERIFICATION]**

OR if volunteering redirects to another website:
3. User selects "Volunteer" (or equivalent CTA)
   * System displays a notice that volunteering will redirect the user to another website 
4. User clicks on redirect link and completes steps there.

## Inner Monologue
- “I’m already signed in, so this should be fast—I just want to lock in my spot.”
- "A lot of these seem like they are looking for a long term commitment, I just want a one-off project."
- "This project looks interesting!"
- “Do I need to pick a shift, or am I volunteering for the whole event?”
- “Am I fully confirmed, or is this pending approval?”
- “I want to be sure I know where to go and what to bring, so I don’t mess this up.”
- "Oh this is redirecting me to another site? I hope that it is secure"

*Primary friction points*
- Unclear whether the signup is instant confirmation or a request requiring approval. **[NEEDS VERIFICATION]**
- Uncertainty about where to view or manage the commitment later.

## End State (Success Indicators)
- **Goal Achievement:** The volunteer commitment is successfully created for the selected project.
- **System Integrity:** The commitment is correctly associated with the signed‑in user and the correct project, date, and shift (if applicable). **[NEEDS VERIFICATION]**
- **Next Steps Clarity:** The user knows when and where to show up, who to contact, and where to view or manage their commitment later.

## Alternative Paths / Edge Cases
- If the project is full, system displays a “Full” or “Waitlist” state and offers alternatives. **[NEEDS VERIFICATION]**
- If the project requires approval, system shows a “Pending approval” status and explains what happens next. **[NEEDS VERIFICATION]**
- If required profile information is missing, system prompts the user to complete it before allowing signup. **[NEEDS VERIFICATION]**
